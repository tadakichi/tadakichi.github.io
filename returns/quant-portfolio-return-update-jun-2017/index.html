Quant Portfolio Return Update Jun 2017

August 27, 2017

Hi, all. In here I would like to update the return of my small size portfolio which I have started investing recently.

This portfolio is 3-month rebalancing-basis, so every month I invest 1⁄3 of my total liquidity, and after 3 months rebalance each of them.

The investing strategy is simple; 1) leave out too low market-cap stocks from the whole of Korean stock market and filter by PSR, ROE basis, 2) and invest 10 stocks of highest rank by simple momentum, volatility, and PSR variable basis.

Expecting 31% annual return, but very low Sharpe ratio. Maybe that’s because there are only 10 stocks in the portfolio and the model has flawed. It should be handled by applying like dual momentum skim on the portfolio itself induced by the model or changing cash proportion every month.

And differences of returns are quite high. Maybe because of not applying timing of the financial states, but also the high influence of single stock and the problem with the model.

Looking at invested stocks, there are many many repeatedly shown stocks. It seems that it is because few low value and volatility stocks appear many times. It should be reduced by applying another variable or so.

Hit ratio or max, min returns are a 1-month basis, so by 3-month as rebalancing basis, it should be higher.

It seems like simulation period is too short. For now, when extracting stocks use daily price and market-cap, so long simulation period causes delayed simulation time, lots of memory consumption. And personally, I think that using data that was too long ago is rather meaningless, especially Korean market. Anyway, those problem causes so called ‘data snooping’ or ‘data mining’ issues, but I think try-and-error method fits for me.

The data over the end of 2015 is for validation. It seems too short but will get bigger as time goes by. I just use it to check the model works well for an out-of-sample period.

For now, it takes approximately 180 seconds to simulate. It works as single thread scheme, so parallelizing will make time much shorter.
